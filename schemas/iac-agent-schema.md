# META-PROMPT: Infrastructure-as-Code (IaC) & Deployment Schema
**Purpose:** This schema instructs an AI agent on how to generate the `## Infrastructure-as-Code (IaC) & Deployment Standards` section for a repository-specific `AGENTS.md` file.
**Usage:** Feed this into the orchestrator alongside the repository context to lock down resource lifecycle compliance, eliminate configuration drift, and guarantee script idempotency.

## SYSTEM INSTRUCTIONS FOR AGENTS.MD GENERATOR

When generating the `## Infrastructure-as-Code (IaC) & Deployment Standards` section of the target repository's `AGENTS.md` file, you must strictly follow the execution steps below. Output only terse, machine-readable constraints.

### STEP 1: Execute Infrastructure Tooling Assessment
Analyze the target repository to detect existing infrastructure patterns, API orchestration scripts, and configuration formats. Inject targeted directive paths into the final `AGENTS.md`:

* **IF [Bicep / ARM Templates present]:** Designate Bicep as the source of truth for cloud topology.
* **IF [Terraform / HCL present]:** Designate Terraform as the absolute provider state mechanism.
* **IF [Imperative Orchestration scripts present (e.g., PowerShell targeting Skillable Studio APIs or custom Azure SDK pipelines)]:** Enforce programmatic API integration contracts and strict session boundary tracking.

### STEP 2: The Idempotency & State Mandate
Inject absolute rules to prevent agents from creating breaking side-effects or resource duplication across environments:

* **The Idempotency Clause:** Every deployment script, lab provisioning automation, or resource runner generated by the agent MUST be natively idempotent. Running the script $N$ times against an identical target state must yield the exact same outcome as running it once, without throwing errors or generating duplicate resources.
* **Declarative Over Imperative:** The agent is strictly forbidden from using raw imperative provisioning commands (e.g., CLI commands like `az container create` or PowerShell commands like `New-AzContainerApp`) if a declarative resource schema (Bicep/ARM/Terraform) can achieve the same target configuration.
* **Pre-Flight Validation:** All deployment modules must execute a validation pass or state check (e.g., `Test-AzResourceGroupDeployment` or API `GET` confirmation) before attempting mutation operations.

### STEP 3: Enterprise Tagging & Compliance Contracts
Ensure that any infrastructure scaffolded or modified by an agent carries complete metadata for tracking, billing, and lifecycle management. Inject these metadata rules:

* **Mandatory Tagging Block:** Every cloud resource declaration must include a standardized tag block. The agent must fail the execution if these fields are omitted or left blank:
    * `Environment`: (e.g., Dev, Test, Prod, Lab-Instance)
    * `Owner`: (The triggering agent principal or orchestration module name)
    * `BillingCode` / `Project`: (The specific enterprise cost tracker allocation)
* **WAF (Well-Architected Framework) Compliance:** Generated topologies must default to secure-by-design patterns: disabled public endpoints where internal virtual networks exist, explicit diagnostic logging enabled, and least-privilege managed identities assigned.

### STEP 4: Ephemeral Resource & Cleanup Boundaries
For dynamic, automated architectures (such as lab images, container pipelines, or on-demand streaming environments), inject strict resource termination rules:

* **The Lifecycle Expiry Rule:** Any dynamically spawned environment or testing resource group MUST be assigned a time-to-live (TTL) tag or an explicit decommissioning contract.
* **Deterministic Teardown Blocks:** Every provisioning runbook must have a corresponding, verified destruction or cleanup routine (`Undo-`, `Remove-`, or `Destroy`). This cleanup routine must explicitly scrub dependent network interfaces, public IPs, and storage targets to eliminate orphaned resource spend.
